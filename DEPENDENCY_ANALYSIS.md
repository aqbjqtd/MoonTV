# MoonTV 依赖分析与镜像优化报告

## 📊 项目依赖现状分析

### 基础统计

- **项目名称**: MoonTV
- **包管理器**: pnpm v10.14.0
- **依赖总数**: 56 个 (生产: 28, 开发: 28)
- **当前镜像大小**: 1.37GB
- **node_modules 大小**: 798.2MB (58%总体积)

### 包管理器对比

| 指标          | pnpm              | npm               | 差异           |
| ------------- | ----------------- | ----------------- | -------------- |
| Lock 文件大小 | 440KB (13,533 行) | 676KB (18,898 行) | +54%           |
| 安装速度      | 快 (并行安装)     | 中等              | pnpm 快 2-3 倍 |
| 磁盘效率      | 高 (硬链接)       | 中等              | pnpm 更优      |
| Docker 兼容性 | 复杂              | 简单              | npm 更优       |

## 🎯 主要大体积依赖分析

### 核心媒体播放依赖 (预计占用~400MB)

1. **@vidstack/react** 1.12.13 - 现代视频播放器框架
2. **framer-motion** 12.18.1 - 高性能动画库
3. **artplayer** 5.2.5 - HTML5 视频播放器
4. **hls.js** 1.6.6 - HLS 流媒体播放库
5. **swiper** 11.2.8 - 滑动组件库

### React 生态系统 (预计占用~200MB)

1. **next** 14.2.23 - Next.js 框架
2. **react/react-dom** 18.2.0 - React 核心
3. **@headlessui/react** 2.2.4 - 无头 UI 组件

### UI/样式依赖 (预计占用~100MB)

1. **tailwindcss** 3.4.17 - CSS 框架
2. **@heroicons/react** 2.2.0 - 图标库
3. **lucide-react** 0.438.0 - 图标库
4. **react-icons** 5.4.0 - React 图标

## 🔍 发现的问题

### 1. 依赖版本冲突

- **@cloudflare/next-on-pages** 需要 next@">=14.3.0"
- **当前 next 版本**: 14.2.23
- **影响**: 构建时需要使用 --legacy-peer-deps

### 2. 依赖冗余

- 同时使用多个图标库 (@heroicons/react, lucide-react, react-icons)
- 多个媒体播放器库 (artplayer, @vidstack/react, hls.js)

### 3. pnpm 在 Docker 中的复杂性

- Corepack 配置增加复杂性
- Store 管理在容器中较复杂
- 构建时间较长

## 🚀 优化方案建议

### 方案 1: 切换到 npm (推荐)

**优势**:

- 简化 Docker 构建流程
- 减少配置复杂度
- 更好的容器兼容性
- 预计减少 5-10%构建时间

**实施步骤**:

1. 使用已有的 package-lock.json
2. 修改 Dockerfile 使用 npm
3. 采用--legacy-peer-deps 解决兼容性

### 方案 2: 依赖整合优化

**预期效果**: 减少 200-300MB 体积

**具体建议**:

1. **图标库统一**: 移除@heroicons/react 和 react-icons，仅保留 lucide-react
2. **媒体播放器整合**: 评估是否可以统一使用@vidstack/react
3. **动画库优化**: 考虑用 CSS 动画替代部分 framer-motion 功能

### 方案 3: 构建策略优化

**多阶段构建优化**:

- 分离生产和开发依赖安装
- 更激进的文件清理策略
- 使用.npmignore 优化构建上下文
